⏺ Short answer: not directly. model_gen has no --radius/--center/--bbox flag —
  the area covered is implicitly defined by whatever boundary shapefile you pass
   to --shape.

  What it does have:
  - --shape <file> / --dbf <file> — the boundary polygons drive everything. OSM
  data and buildings outside the shapefile bounds are effectively ignored, and
  PUMAs are auto-clipped to the shapefile bounds (ModelGenerator.cpp:1895-1905).
  - --puma-ring <id> / --pop-ring <id> — restrict drawing to one ring, but these
   are visualization filters, not generation filters.

  So the standard workaround is to shrink the inputs before invoking model_gen:

  1. Make a small boundary shapefile — use QGIS or ogr2ogr to cut a
  few-mile-radius polygon (or a small bbox) out of the full city boundary
  shapefile. Pass that to --shape/--dbf. PUMA filtering will follow
  automatically.
  2. Pre-clip the OSM file — use osmium extract with either a bbox (osmium 
  extract -b minlon,minlat,maxlon,maxlat full.osm -o small.osm) or the same
  polygon (osmium extract -p area.geojson ...). Smaller OSM = much faster run
  and avoids loading a city's worth of ways.
  3. PUMS/PUMA data can stay the same — only the PUMAs intersecting your small
  shapefile will be retained.

  radius around a lat/lon in Chicago, plus the matching model_gen invocation?

⏺ Here's a worked example: 3-mile radius around the Chicago Loop (41.8781°N, 
  -87.6298°W). The original tests/chicago/run.sh:20 is the baseline I'm
  shrinking.

  1. Build a clip polygon (proper buffered circle, not a bbox)

  A degree-based buffer would stretch east-west since Chicago is at ~42°
  latitude. Buffer in a projected CRS (meters), then reproject back to WGS84.
  UTM zone 16N (EPSG:26916) is right for Chicago. 3 miles = 4828 m.

  # From tests/chicago/
  mkdir -p clip
  CENTER_LAT=41.8781
  CENTER_LON=-87.6298
  RADIUS_M=4828   # 3 miles

  # Make a WGS84 point, project to UTM 16N, buffer 4828 m, reproject to WGS84
  ogr2ogr -f "ESRI Shapefile" clip/center.shp \
    -dialect SQLite -sql "SELECT GeomFromText('POINT(${CENTER_LON}
  ${CENTER_LAT})', 4326) AS geom" \
    -a_srs EPSG:4326 /vsistdin/?dummy.csv 2>/dev/null

  # Easier: one shot with GDAL >= 3.x via ogr2ogr + a tiny CSV input
  cat > clip/center.csv <<EOF
  id,wkt
  1,POINT(${CENTER_LON} ${CENTER_LAT})
  EOF

  ogr2ogr -f "ESRI Shapefile" clip/center_utm.shp clip/center.csv \
    -oo GEOM_POSSIBLE_NAMES=wkt -a_srs EPSG:4326 -t_srs EPSG:26916

  ogr2ogr -f "ESRI Shapefile" clip/circle_utm.shp clip/center_utm.shp \
    -dialect SQLite -sql "SELECT ST_Buffer(geometry, ${RADIUS_M}) AS geometry
  FROM center_utm"

  ogr2ogr -f "ESRI Shapefile" clip/circle_wgs84.shp clip/circle_utm.shp \
    -t_srs EPSG:4326

  That gives you clip/circle_wgs84.shp (+ .dbf, .shx, .prj) — feed this to
  --shape/--dbf.

  2. Pre-clip the OSM file with osmium

  Extract a bbox slightly larger than the circle so OSM ways spanning the edge
  stay connected, or use the polygon directly:

  # Polygon-based extract (preferred — matches the circle exactly)
  ogr2ogr -f GeoJSON clip/circle_wgs84.geojson clip/circle_wgs84.shp
  osmium extract -p clip/circle_wgs84.geojson Chicago_2023.osm \
    -o Chicago_loop3mi.osm --overwrite

  # OR quick bbox alternative (looser, ~5% bigger area):
  # osmium extract -b -87.6883,41.8346,-87.5713,41.9216 Chicago_2023.osm \
  #   -o Chicago_loop3mi.osm --overwrite

  3. Run modelgen against the clipped inputs

  Same flags as run.sh:20, just swap the shapefile and OSM:

  /usr/bin/time ../../modelgen \
    --shape clip/circle_wgs84.shp \
    --dbf   clip/circle_wgs84.dbf \
    --pop-gis lspop2011/w001001.adf \
    --osm-xml Chicago_loop3mi.osm \
    --adjust-model chicago_adjustments.txt \
    --xfig chicago_loop3mi.fig \
    --scale 16384000 \
    --out-model chicago_loop3mi_model.txt \
    --pums-h pums/illinois_2021_5years/psam_h17.csv \
    --pums-p pums/illinois_2021_5years/psam_p17.csv \
    --puma-shp pums/ipums_puma_2010/ipums_puma_2010_WGS84.shp \
    --puma-dbf pums/ipums_puma_2010/ipums_puma_2010_WGS84.dbf \
    --pums-h-cols SERIALNO \
    > bld_info_loop3mi.txt

  The PUMS/PUMA files stay full-state — ModelGenerator.cpp:1895-1905 will
  auto-restrict to PUMAs intersecting the small circle shapefile.

  Caveats

  - The lspop2011 raster is national; no clipping needed, modelgen samples only
  inside the shape.
  - chicago_adjustments.txt lists per-building tweaks for the whole city — most
  lines will simply not match anything inside the 3-mile circle, which is
  harmless. If you want to be safe, omit --adjust-model for the small run.
  - If the circle straddles two PUMAs, both PUMS samples are used. To force one,
   add --puma-ring <id> (but that's a drawing filter, so verify by reading the
  bld_info log).
