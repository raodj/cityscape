  Required inputs for a new city (Philadelphia)

  1. City boundary shapefile (--shape, --dbf)

  A polygon shapefile defining the city limits. You need all four sidecar files
  (the tool only takes .shp + .dbf as args, but .shp needs the others to open):
  - philadelphia.shp — geometry
  - philadelphia.dbf — attributes
  - philadelphia.shx — index
  - philadelphia.prj — projection (must be WGS 84 / EPSG:4326)

  Source: city open-data portal (OpenDataPhilly has a "City Limits" layer).
  Reproject to WGS84 with ogr2ogr -t_srs WGS84 out.shp in.shp if it isn't
  already.

  2. OSM XML (--osm-xml)

  - Philadelphia.osm — buildings + streets

  Source: https://download.bbbike.org/osm/bbbike/ (pick or request a
  Philadelphia extract; download the .osm.gz and gunzip it).

  3. PUMS microdata (--pums-h, --pums-p) — Pennsylvania, since Philadelphia is 
  in PA

  - psam_h42.csv — household records (state code 42 = PA)
  - psam_p42.csv — person records

  Source: https://www.census.gov/programs-surveys/acs/microdata/access.html.
  Pick a vintage (e.g., ACS 2022 5-year).

  4. PUMA shapefile (--puma-shp, --puma-dbf) — same vintage as PUMS

  - puma_pa.shp
  - puma_pa.dbf
  - puma_pa.shx, puma_pa.prj (sidecars)

  Source: https://www.census.gov/cgi-bin/geo/shapefiles/index.php (state-level
  PUMA layer for Pennsylvania). Reproject to WGS84:
  ogr2ogr -t_srs WGS84 puma_pa_WGS84.shp puma_pa.shp
  Per the README, use the same year for PUMS and PUMA.

  5. Model adjustments file (--adjust-model, optional but recommended)

  - philadelphia_adjustments.txt — you author this. Copy
  tests/chicago/chicago_adjustments.txt and tune HomeBuildingTypes,
  ApartmentBuildingTypes, IgnoreBuildingTypes, DefaultApartmentLevels for
  Philly's OSM tagging quirks. Also lists OSM building IDs to ignore.

  6. Optional: LandScan population GIS (--pop-gis)

  - lspop20XX/w001001.adf (plus the rest of the ESRI grid sidecars: .adf,
  dblbnd.adf, hdr.adf, etc.)

  LandScan was dropped in the Jan 2025 Chicago run (see run.sh:24) — runs are
  slower without it but it's no longer required.

❯ can you explain what each #SBATCH comment does                                

⏺ Sure — here's what each line in the script controls. SLURM treats any #SBATCH
  line at the top of the script as a job-submission flag (everything after
  #SBATCH is identical to a command-line flag for sbatch).

  #!/bin/bash
  Standard shebang — tells the system to run the script with bash once SLURM
  hands it to a compute node. Not SLURM-specific.

  #SBATCH --job-name=philly_modelgen
  Name shown in squeue, sacct, and email subjects. Cosmetic, but makes your jobs
   easy to find when you have several running.

  #SBATCH --time=48:00:00
  Wallclock limit in HH:MM:SS. SLURM kills the job at this point even if it
  isn't done. Set it generously but not absurdly — shorter requests often start
  sooner because the scheduler can backfill them.

  #SBATCH --nodes=1
  How many physical compute nodes to allocate. modelgen is a single-process
  OpenMP program, so 1 node is correct. (You'd raise this only for MPI jobs.)

  #SBATCH --ntasks-per-node=1
  How many processes (think MPI ranks) to launch per node. modelgen is one
  process, so 1.

  #SBATCH --cpus-per-task=8
  How many CPU cores to give that one process. This is what enables OpenMP
  parallelism — modelgen has #pragma omp parallel for loops (DrawBuildings,
  LinearWorkBuildingAssigner), so more cores = faster. 8 is a sensible default;
  bump up if your cluster has fat nodes.

  #SBATCH --mem=64G
  Total RAM for the job. OSM XML + PUMS for a city the size of Philly is
  memory-hungry; 64G gives headroom. If the job OOMs you'll see it in the .err
  file and you raise this.

  #SBATCH --mail-user=stojansz@miamioh.edu
  #SBATCH --mail-type=END,FAIL
  Email notifications. END,FAIL = send mail when the job finishes successfully
  or fails. Other options: BEGIN, ALL, TIME_LIMIT_80, etc.

  #SBATCH -A PMIU0110
  Account / allocation code to charge the compute hours against. PMIU0110 is the
   same one used in model_gen/schedule.sh — likely your OSC project allocation.
  If you're submitting to a different cluster, change this.

  #SBATCH -o philly_modelgen_%j.out
  #SBATCH -e philly_modelgen_%j.err
  Where stdout and stderr go. %j is substituted with the SLURM job ID, so each
  run produces uniquely-named log files instead of overwriting the previous run.
   Without these, both streams default to slurm-<jobid>.out in the submit
  directory.

  ---
  One thing worth knowing about the body of the script

  export OMP_NUM_THREADS=${SLURM_CPUS_PER_TASK:-1}
  This isn't a SLURM directive — it's a regular shell command that runs on the
  compute node. SLURM exports SLURM_CPUS_PER_TASK automatically (matches
  whatever you put in --cpus-per-task), and OpenMP reads OMP_NUM_THREADS to
  decide how many threads to spawn. Wiring them together means if you change
  --cpus-per-task=16 later, OpenMP picks that up automatically — no second edit
  needed.

❯ What does "export OMP_NUM_THREADS=${SLURM_CPUS_PER_TASK:-1}" do?              
⏺ It wires SLURM's CPU allocation into OpenMP. Breaking it down:

  OMP_NUM_THREADS

  An environment variable that OpenMP reads at runtime to decide how many 
  threads to spawn in parallel regions (the #pragma omp parallel for loops in
  modelgen). If unset, OpenMP defaults to "however many cores the OS reports,"
  which on a shared cluster node is usually wrong — it'd try to use all 128
  cores even though SLURM only gave you 8.

  ${SLURM_CPUS_PER_TASK:-1}

  This is bash parameter expansion with a default.

  - SLURM_CPUS_PER_TASK is a variable SLURM automatically sets inside your job,
  matching whatever you requested with #SBATCH --cpus-per-task=8. So inside the
  job it equals 8.
  - :-1 means "if this variable is unset or empty, use 1 instead." The colon
  makes the fallback fire on empty values too, not just unset ones.

  So if you ever run the script outside SLURM (e.g., locally for testing) where
  SLURM_CPUS_PER_TASK doesn't exist, OpenMP falls back to single-threaded
  instead of crashing or going haywire.

  export

  Without export, the variable would only exist in the current shell. export
  pushes it into the environment of child processes — i.e., modelgen, when it
  runs, will see OMP_NUM_THREADS and honor it.

  Put together

  "Tell OpenMP to use exactly as many threads as SLURM gave me cores for,
  falling back to 1 if SLURM isn't in the picture."

  The payoff: if you later edit #SBATCH --cpus-per-task=16, this line picks it
  up automatically — no second place to update.

  What's happening

  PUMS.cpp:72 throws when the column name you told it to look for isn't present

  Why the column is missing

  Different PUMA shapefile sources use different column names:

  Source / vintage: Census TIGER, 2010 vintage
  DBF column: PUMACE10
  matching PUMS CSV column: PUMA10
  ────────────────────────────────────────
  Source / vintage: Census TIGER, 2020 vintage
  DBF column: PUMACE20
  matching PUMS CSV column: PUMA20
  ────────────────────────────────────────
  Source / vintage: IPUMS-prepared (what Chicago test uses)
  DBF column: PUMA
  matching PUMS CSV column: PUMA

  You almost certainly downloaded a TIGER shapefile, which doesn't have a bare
  PUMA column.

  Fix — check, then pass the right column names

  First, look inside your DBF to confirm the column name:

  cd tests/philadelphia
  # Use dbfdump if you have it, or ogrinfo, or python:
  python3 -c "from dbfread import DBF; 
  print(DBF('pums/<your_puma>.dbf').field_names)"
  # or:
  ogrinfo -al -so pums/<your_puma>.shp | grep -E "^[A-Z]"

  shapefile:

❯ can you explain what ogr2ogr and what ogrinfo do, and what WGS84 means?       

⏺ ogr2ogr and ogrinfo

  Both come from GDAL/OGR — the open-source geospatial data library that
  underlies QGIS, PostGIS, and basically every GIS tool. "OGR" was originally
  the vector-data half of GDAL (the "G" half handled raster); they've since
  merged, but the two command names stuck.

  ogrinfo — read-only inspector

  Prints metadata about a vector dataset (shapefile, GeoJSON, GeoPackage,
  PostGIS table, etc.) without modifying it. Think of it as file + head for
  geospatial vector data.

  Common flags:
  - -al — all layers (shapefiles only have one, but GeoPackages can have many)
  - -so — summary only (don't dump every feature's attributes — just the schema,
   CRS, extent, count)

  So ogrinfo -al -so foo.shp answers: what columns does this have, what 
  coordinate system, what's the bounding box, how many features? That's exactly
  what you ran to discover PUMACE20 and NAD83.

  Without -so it would dump all 95 PUMA polygons' attribute rows — useful for
  spot-checking, overwhelming for big files.

  ogr2ogr — converter / transformer

  Reads from one vector dataset and writes to another, optionally transforming
  on the way. The pattern is always:

  ogr2ogr [options] OUTPUT INPUT

  (Output first — easy to get backwards.) Common uses:

  - Reproject: ogr2ogr -t_srs EPSG:4326 out.shp in.shp — -t_srs = "target
  spatial reference system"
  - Convert format: ogr2ogr out.geojson in.shp — format inferred from extension
  - Filter rows: ogr2ogr -where "STATEFP20='42'" out.shp in.shp
  - Clip to a box: ogr2ogr -clipsrc -75.3 39.8 -74.9 40.2 out.shp in.shp

  For your case only -t_srs EPSG:4326 matters — it reads the input's declared
  CRS from the .prj file, computes the math to transform each vertex into the
  target CRS, and writes a new shapefile (with a new .prj declaring EPSG:4326).

  It does not mutate the input. Always produces new files.

  ---
  What "WGS84" means

  WGS84 = World Geodetic System 1984. It's the global reference frame the GPS
  satellite network broadcasts in, and it has two pieces that work together:

  1. A datum — a model of Earth's shape

  Earth isn't a perfect sphere; it's lumpy and slightly squashed at the poles. A
   datum is a mathematical reference ellipsoid (and an origin point) that
  approximates Earth so you can assign coordinates to locations. WGS84 picks a
  specific ellipsoid (semi-major axis 6,378,137 m, flattening 1/298.257223563)
  centered on Earth's center of mass.

  The other datum you saw — NAD83 (North American Datum 1983) — uses a slightly
  different ellipsoid optimized for North America. The two differ by ~1–2 meters
   in the continental US. For modeling buildings and intersections, that
  matters; for navigating a car, not really.

  2. A coordinate system — how you label points on that ellipsoid

  WGS84 the datum is most commonly paired with geographic (lat/lon, in degrees)
  coordinates. That combination has an official identifier: EPSG:4326.

  - "EPSG" = European Petroleum Survey Group, who maintain the universal catalog
   of CRS codes.
  - 4326 is the registry number for "WGS84, lat/lon, degrees."

  So when GDAL prints GEOGCRS["WGS 84"... ID["EPSG", 4326]], it's saying:
  geographic coordinates (lat/lon in degrees), on the WGS84 ellipsoid.

  Why it matters here

  Three competing things in your model:

  - OSM publishes everything in EPSG:4326 — fixed, can't change.
  - TIGER PUMA shapefiles ship in NAD83. Visually identical, numerically ~1m
  off.
  - Local shapefiles (city boundaries) often ship in a projected CRS like
  EPSG:2272 / NAD83 Pennsylvania South (US feet), where coordinates are meters 
  or feet from a local origin, not lat/lon. Values look like (2660000, 210000)
  instead of (-75.16, 39.95).

  modelgen does naive lat/lon math — it assumes every input is EPSG:4326 and
  compares coordinates directly. Mix a State Plane boundary (millions of feet)
  with an OSM dataset (degrees), and the boundary "lives" on Mars relative to
  OSM; nothing intersects. Mix NAD83 with WGS84 and everything lines up almost
  perfectly but PUMA edges land 1–2 m off — most buildings get assigned
  correctly, edge cases don't.

  Reprojecting to EPSG:4326 with ogr2ogr -t_srs EPSG:4326 is what gets them all
  speaking the same language.
